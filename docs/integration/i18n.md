# Localization (i18n)

L10n guide for `nsg_messenger` SDK.

## Overview

The SDK ships **RU** and **EN** strings baked into the package. Localization is ARB-based via `flutter gen-l10n` — source files live in `sdk/nsg_messenger/lib/src/i18n/l10n/app_*.arb`, generated bindings in `sdk/nsg_messenger/lib/src/i18n/generated/nsg_l10n.dart`.

The host app must register the delegate on its `MaterialApp` for the SDK widgets to render localized strings:

```dart
import 'package:nsg_messenger/nsg_messenger.dart';

MaterialApp(
  localizationsDelegates: NsgL10n.localizationsDelegates,
  supportedLocales: NsgL10n.supportedLocales,
  // ...
);
```

`NsgL10n.localizationsDelegates` is a convenience list bundling the SDK delegate plus `GlobalMaterialLocalizations.delegate`, `GlobalCupertinoLocalizations.delegate`, and `GlobalWidgetsLocalizations.delegate`. If the host needs to compose with additional delegates, append to this list (or substitute the bare `NsgL10n.delegate`).

Default locale resolves from the OS via Flutter's standard `Localizations` mechanism; override at SDK init via the `locale:` parameter:

```dart
await NsgMessenger.init(
  apiBaseUrl: '...',
  authTokenProvider: MyAuthProvider(),
  locale: const NsgMessengerLocale(locale: Locale('en')),
);
```

## Adding a new locale

To add (e.g.) German:

1. **Copy the template.** Duplicate `sdk/nsg_messenger/lib/src/i18n/l10n/app_en.arb` to `app_de.arb`. Keep the `@@locale` header set to the new code.
2. **Translate values.** Keep all keys identical to `app_en.arb`. Drop the `@-prefixed` metadata blocks in the localized file (they live only in the template). Only the values change.
3. **Regenerate bindings.** From `sdk/nsg_messenger/`:
   ```bash
   flutter gen-l10n
   ```
   This rewrites `lib/src/i18n/generated/nsg_l10n.dart` + per-locale subclasses (`nsg_l10n_de.dart`, etc) and updates `NsgL10n.supportedLocales`.
4. **Validate.** The repo's lefthook pre-commit hook runs `tool/hooks/l10n_validate.sh`, which checks for missing keys, stray placeholders, and stale generated files. Run it manually before pushing:
   ```bash
   bash tool/hooks/l10n_validate.sh
   ```
5. **Submit PR.** Note the locale + completeness percentage in the PR description so this doc's "Built-in locales" table can be updated.

## Overriding individual strings

**Currently NOT supported.** The original TASK22 plan included `NsgMessengerTheme.l10nOverrides: Map<String, String>` for host apps that want to relabel SDK strings (e.g. "Chats" → "Conversations") without forking the SDK, but that's a **Phase 3** item — not in this MVP commit.

Workaround until then: if you need to relabel a string, file an issue against this repo with the key + suggested wording. We'll either expand the ARB (if it's a genuine translation gap) or accelerate the `l10nOverrides` feature.

## System locale resolution

`NsgMessengerLocale.resolveFromSystem()` is the helper the SDK uses internally. At the current commit it always returns `Locale('ru')` regardless of the OS locale (a stub from TASK22 Chunk 1; the real implementation will look at `WidgetsBinding.instance.platformDispatcher.locale` and match against `NsgMessengerLocale.supported`). In practice this rarely matters because:
- The host's `MaterialApp.supportedLocales` + `NsgL10n.delegate` drive actual string resolution through Flutter's standard `Localizations` lookup — the SDK reads `NsgL10n.of(context)`, which respects the active locale negotiated by Flutter.
- You can always override explicitly with `init(locale: NsgMessengerLocale(locale: Locale('en')))`.

## Date formatting

The SDK uses `intl.DateFormat` (from `package:intl`) for timestamps and `timeago` for relative deltas (e.g. "5 min ago"). Both packages key off the active Flutter `Locale`, so once `MaterialApp.locale` / `NsgL10n.delegate` is wired correctly, no extra setup is required.

If your host app changes locale at runtime (e.g. a language picker), make sure to also call `Intl.defaultLocale = newLocaleString` so any non-widget date formatting in shared code follows along.

## Server-side i18n

Push notification bodies are localized **server-side** based on `DeviceRegistration.locale`, which the SDK populates from `DeviceInfo.locale` returned by the `PushTokenProvider`. The host customer doesn't need to do anything — adding a new mobile locale to the SDK doesn't automatically translate push bodies (that's a separate server-side ARB / template rollout in the backend repo).

If you add a new locale to the SDK but don't see push notifications in that language, check that the backend's push template catalog has the same locale registered.
